🧠 Core Concepts

🗂

Namespaces

A Namespace is a virtual cluster within your K8s cluster. Resources (Pods, Services, ConfigMaps) in one namespace are isolated from another by default. One cluster can host dev, staging, prod workloads simultaneously — each in its own namespace with its own config and access controls.

Interview: "Namespaces provide logical isolation. The same cluster runs dev and prod, but dev cannot accidentally talk to prod services because they are in different namespaces with network policies applied."

⚙️

ConfigMap

A ConfigMap stores non-sensitive configuration as key-value pairs. Your pod reads from it at runtime — either as environment variables or mounted as files. Change the ConfigMap → restart the pod → new config applied. No image rebuild needed.

Without ConfigMap: APP_ENV=production is hardcoded in your deployment.yaml. To change it for dev, you edit the YAML. With ConfigMap: the YAML stays the same across environments, only the ConfigMap changes. This is the correct pattern.

🔐

Secret

A Secret stores sensitive data (passwords, tokens, keys). Values are base64-encoded (not encrypted by default — encryption at rest needs additional configuration). Kubernetes restricts which pods can mount which secrets via RBAC. Never put passwords in ConfigMaps or in YAML files.

Secrets vs ConfigMaps: same mechanics, different intent. K8s can apply additional controls to Secrets (audit logging, encryption, RBAC restrictions). Use ConfigMap for config, Secret for credentials. Never the other way around.

🌍

Environment Variables from ConfigMap/Secret

Two ways to inject: envFrom — inject all key-value pairs from a ConfigMap/Secret as env vars. env.valueFrom — inject specific keys selectively. envFrom is simpler. valueFrom gives finer control — useful when a ConfigMap has 20 keys but your pod only needs 3.

In Spring Boot: the injected environment variables override application.properties values. SPRING_DATASOURCE_URL env var overrides spring.datasource.url in the properties file. Spring Boot convention automatically maps these.

🔄

ConfigMap as Volume Mount

Instead of env vars, a ConfigMap can be mounted as a file inside the container. Key = filename, value = file content. This is used for config files (nginx.conf, application.properties, log4j.xml). The pod reads the file at the path you specify.

For your Spring Boot app: mount application.properties from a ConfigMap. Change ConfigMap → restart pod → new properties file. No image rebuild — this is how configuration changes are managed in production without redeployments.

📋

Resource Quotas + LimitRange

Namespaces can have ResourceQuota (maximum total CPU/memory the namespace can use) and LimitRange (default and maximum per-pod limits). This prevents one team's dev workload from starving another team's prod workload in a shared cluster.

In your learning setup you won't hit quotas. In an enterprise shared cluster, quotas are mandatory. Understanding them is an interview differentiator — shows you think about multi-tenancy.

⚠ ConfigMap + Secret Mistakes
✗
Putting passwords in ConfigMaps instead of Secrets

ConfigMaps have no access controls, no audit logging, and are shown in plain text by kubectl get configmap -o yaml. Anyone with kubectl read access sees them.

✓ Fix: Rule — if it is sensitive (password, token, key, certificate), it goes in a Secret. If it is configuration (log level, port, env name), it goes in ConfigMap.
✗
Committing secret.yaml with real values to Git

Your secret.yaml has base64-encoded values that decode trivially. Committing to Git = credentials in version history forever.

✓ Fix: Add secret.yaml to .gitignore. Commit secret.yaml.example with placeholder values. In production use Sealed Secrets or HashiCorp Vault to manage secrets safely in Git.
✗
Forgetting to restart deployment after ConfigMap change

ConfigMap changes are not detected by running pods. The old value stays until pods restart. This catches people off-guard — they edited the ConfigMap, app still uses old value, confusion follows.

✓ Fix: kubectl rollout restart deployment/NAME -n NAMESPACE after every ConfigMap change. Or use a tool like Reloader (watches ConfigMaps and auto-restarts deployments).

INT Q →"What is the difference between a Kubernetes ConfigMap and a Secret?"
SeniorAnswer: "Both ConfigMap and Secret store key-value data that pods consume as environment variables or mounted files. The difference is intent and access control. ConfigMaps store non-sensitive configuration — log levels, port numbers, feature flags — and are stored in plain text in etcd. Secrets store sensitive data — passwords, API tokens, TLS certificates — and are stored base64-encoded in etcd. Base64 is not encryption, it is encoding. For real security you need etcd encryption at rest enabled and RBAC policies restricting which service accounts can read which secrets. In production I would also use Sealed Secrets or HashiCorp Vault so that secret values are never committed to Git in any form. The practical rule I follow: if you would be embarrassed if a colleague saw the value in a kubectl get configmap output, it belongs in a Secret."
⚡ Day 15 Quiz — Pass = 4/5
Q1. You update a ConfigMap value. The pod is still using the old value. Why?
A) ConfigMap changes require deleting and recreating the namespace
B) Running pods do not detect ConfigMap changes automatically — you must restart the deployment
C) Only Secrets update automatically
D) You need to re-apply the deployment YAML
Pods read ConfigMap values at startup — they are injected as environment variables when the container starts. A ConfigMap change does not trigger pod restarts. You must trigger a rolling restart: kubectl rollout restart deployment/NAME -n NAMESPACE. The new pods start with the updated ConfigMap values. Old pods are terminated after new ones are ready.
Q2. What command shows you the current namespace your kubectl context is using?
A) kubectl get namespace current
B) kubectl config view --minify | grep namespace
C) kubectl namespace --current
D) kubectl get context
kubectl config view --minify shows only the current context settings including the active namespace. Always verify your namespace before applying YAMLs — applying to the wrong namespace (e.g., prod instead of dev) is a common and potentially damaging mistake. Alternatively: kubectl config get-contexts shows all contexts with the current one marked.
Q3. Why is base64 encoding in a Secret not the same as encryption?
A) Base64 uses a different algorithm than encryption
B) Base64 is reversible by anyone — echo 'encoded' | base64 -d reveals the original value. Encryption requires a key.
C) Base64 only works for ASCII characters
D) Kubernetes Secrets are actually encrypted by default
base64 is an encoding scheme, not encryption. Any person or program can decode it instantly without a key. kubectl get secret NAME -o yaml shows the base64 value, and base64 -d decodes it. Kubernetes Secrets are only "more secure" than ConfigMaps in that K8s can apply RBAC to restrict who can read them. For actual encryption, you need etcd encryption at rest or an external secrets manager like Vault.
Q4. What is the correct order to apply these Kubernetes resources?
A) Deployment → ConfigMap → Secret
B) ConfigMap → Secret → Deployment
C) Secret → Deployment → ConfigMap
D) Order doesn't matter in Kubernetes
The Deployment references the ConfigMap and Secret by name. If you apply the Deployment first, the pod will fail to start because the referenced ConfigMap or Secret doesn't exist yet. Apply dependencies first: ConfigMap and Secret → then Deployment. Kubernetes will mark the pod as failed with "configmap not found" if you get this wrong.
Q5. What does kubectl rollout status deployment/devops-app -n dev show you?
A) The deployment's resource limits
B) Real-time progress of a rolling update — whether new pods are up and old ones terminated, and if the rollout completed successfully
C) The CPU and memory usage of the deployment
D) The list of ConfigMaps used by the deployment
kubectl rollout status watches the deployment and reports progress: "Waiting for rollout to finish: 1 of 2 updated replicas are available..." until it finishes with "successfully rolled out". It blocks your terminal until complete — useful in scripts and pipelines to wait for a deployment to finish before proceeding. Exit code 0 = success, non-zero = failed rollout.

🧠 Core Concepts — Ingress

🚪

Ingress Controller vs Ingress Resource

Two separate things. The Ingress Controller is a pod running inside the cluster (nginx, Traefik, HAProxy) that watches for Ingress resources and configures routing rules. The Ingress Resource is a YAML file you write that defines the routing rules. Without a controller, an Ingress resource does nothing.

Common mistake: writing an ingress.yaml, applying it, nothing works. Reason: no Ingress controller is installed. In minikube: minikube addons enable ingress installs the controller.

🗺

Path-Based Routing

One Ingress can route different URL paths to different services: /app → devops-app:80, /api → api-service:8080, /admin → admin-service:3000. All on the same hostname, same port 80. The Ingress controller reads the path and forwards to the correct backend service.

This is how microservices architectures work in production — one load balancer IP, different services behind different paths. No separate ports per service.

🌐

Host-Based Routing

Route based on the hostname: app.devops.local → devops-app, admin.devops.local → admin-app. Both resolve to the same Ingress IP. The Ingress controller reads the Host header and routes accordingly. Used when different applications share a cluster but need separate domains.

In production: api.company.com and app.company.com both point to the same ALB/Ingress IP. Routing is done inside the cluster by host header — no separate load balancers needed per app.

🔒

TLS Termination

HTTPS is terminated at the Ingress controller. You provide a TLS certificate as a Kubernetes Secret. The Ingress rule references it. External traffic hits the Ingress on port 443 (encrypted). The Ingress decrypts and forwards to backend services on port 80 (plain HTTP inside the cluster). Backend services don't need to handle TLS.

This is the standard production pattern. Your Spring Boot app does not deal with certificates. The Ingress layer handles all SSL — one certificate, one renewal, many services behind it.

⚖️

ClusterIP Service Type

ClusterIP is the default service type — only reachable from inside the cluster. No external access. This is what you use when the service is accessed via Ingress (Ingress → ClusterIP service → pods). NodePort adds external access directly — bypassing Ingress. LoadBalancer creates a cloud load balancer — expensive and not needed when you have Ingress.

Production pattern: all services use ClusterIP. One Ingress controller is the single entry point. This centralises routing, SSL, rate limiting, and logging in one place.

📋

Ingress Annotations

Annotations customise Ingress controller behaviour: rewrite URL paths, set timeouts, configure rate limiting, force HTTPS redirect. Different controllers use different annotation prefixes. nginx Ingress uses nginx.ingress.kubernetes.io/.

nginx.ingress.kubernetes.io/rewrite-target: / — strips the path prefix before forwarding. Without it, /app/hello forwards /app/hello to the backend which only knows /hello — causing 404s.

⚠ Ingress Mistakes
✗
Using NodePort services with Ingress

Ingress routes to ClusterIP services, not NodePort. Using NodePort with Ingress creates confusion — external traffic hits both the random NodePort directly AND through Ingress, bypassing your routing rules.

✓ Fix: When using Ingress, all services behind it should be type ClusterIP. Only the Ingress controller needs external access.
✗
Forgetting the rewrite-target annotation when using path prefixes

/app/health routed to Spring Boot becomes a 404 because Spring Boot only knows /health — it has no /app prefix. Without rewrite-target the path prefix reaches the backend.

✓ Fix: If routing /app/* to a service whose endpoints start at /, always add the rewrite-target annotation to strip the prefix before forwarding.
✗
Not specifying ingressClassName in ingress.yaml

In newer Kubernetes versions (1.18+), you must specify which IngressClass to use. Without it, no controller picks up the Ingress resource and routing never works — no error, just silence.

✓ Fix: Always add spec.ingressClassName: nginx (or whatever your controller's class name is). Check with kubectl get ingressclass.

INT Q →"Why is NodePort not used in production and what replaces it?"
SeniorAnswer: "NodePort exposes a service on a random port between 30000 and 32767 on every node's IP. In production this has three problems. First, you cannot give users a URL like myapp.company.com:31245 — it's unusable and unmemorable. Second, NodePort provides no SSL termination — you would need to handle TLS in each application. Third, every new service needs a new random port, and managing dozens of ports quickly becomes unmanageable. In production we use an Ingress controller — typically nginx Ingress or a cloud provider's Ingress like AWS ALB Ingress Controller. All services are ClusterIP. The Ingress controller is the single entry point on standard ports 80 and 443. It reads Ingress resources — YAML files defining routing rules — and routes traffic to the correct ClusterIP service based on hostname or path. TLS is terminated at the Ingress layer using a certificate stored in a Kubernetes Secret. Every application team writes their own Ingress resource without touching the central load balancer configuration."
⚡ Day 16 Quiz — Pass = 4/5
Q1. What is the difference between an Ingress Controller and an Ingress Resource?
A) They are the same thing with different names
B) The controller is a running pod that implements routing logic. The resource is a YAML file defining the routing rules. The controller reads and acts on resources.
C) The controller handles HTTPS; the resource handles HTTP
D) Resources are cluster-scoped; controllers are namespace-scoped
This is a common source of confusion. The Ingress Resource (kind: Ingress in YAML) defines WHAT to route and WHERE. It does nothing by itself. The Ingress Controller is a pod running nginx/traefik/etc that watches for Ingress resources and configures itself to implement those rules. Without a controller, Ingress resources are ignored. In minikube, minikube addons enable ingress installs the nginx controller.
Q2. Why do you need the nginx.ingress.kubernetes.io/rewrite-target annotation when routing /app/* to your Spring Boot service?
A) Spring Boot requires this annotation to start
B) Without it, /app/health is forwarded as /app/health to Spring Boot which only has /health — causing a 404
C) The annotation enables HTTPS for that path only
D) It sets the request timeout for that path
When nginx routes /app/health to the backend, it forwards the full path by default. Your Spring Boot app only knows about /health — it has no route registered for /app/health. rewrite-target: /$2 strips the /app prefix and rewrites the path to /health before forwarding. The backend receives /health and responds correctly.
Q3. Where is a TLS certificate stored in Kubernetes for use by an Ingress?
A) In a ConfigMap
B) In a Secret of type kubernetes.io/tls containing tls.crt and tls.key
C) In the Ingress YAML directly as base64
D) In the nginx Ingress controller configuration
TLS certificates are stored as Kubernetes Secrets of type kubernetes.io/tls. The Secret has two keys: tls.crt (the certificate) and tls.key (the private key). Create it with kubectl create secret tls NAME --cert=file.crt --key=file.key. The Ingress resource references this Secret by name in the spec.tls.secretName field. The Ingress controller loads the certificate from the Secret automatically.
Q4. Your Ingress resource is applied but kubectl get ingress shows no ADDRESS. What is the likely cause?
A) The namespace is wrong
B) Either the Ingress controller is not running or minikube tunnel is not running in a separate terminal
C) The service type must be LoadBalancer not ClusterIP
D) ingressClassName is missing from the resource
ADDRESS is assigned by the Ingress controller when it processes the Ingress resource. If the controller pod is not running (check kubectl get pods -n ingress-nginx), no ADDRESS is assigned. In minikube specifically, the ADDRESS also requires minikube tunnel to be running in a separate terminal — without it the IP assignment doesn't propagate. Run minikube tunnel and wait 30 seconds.
Q5. What service type should all services behind an Ingress controller use?
A) NodePort — Ingress requires NodePort to route traffic
B) LoadBalancer — one load balancer per service
C) ClusterIP — internal only, Ingress controller forwards to it from inside the cluster
D) ExternalName — maps to external DNS
ClusterIP services are only reachable from inside the cluster. The Ingress controller runs inside the cluster, so it can reach ClusterIP services directly. Using NodePort or LoadBalancer alongside Ingress creates redundant external access paths — traffic can bypass Ingress by hitting the NodePort directly. ClusterIP + Ingress = one entry point, centralised control of routing, TLS, and rate limiting.
