🧠 Core Concepts

🏗

What Jenkins Actually Is

Jenkins is a self-hosted automation server. It listens for triggers (a push to GitHub, a timer, a manual click) and then runs your pipeline steps in sequence. It is the conductor — Maven, SonarQube, Docker are the musicians. Jenkins tells each one when to play.

Interview: "Jenkins is a CI server that automates the build, test, and deployment pipeline. It's triggered by a webhook from GitHub and orchestrates every downstream tool."

🔌

Jenkins Plugins

Jenkins is minimal out of the box. Almost everything useful — Git integration, Maven support, SonarQube scanning, JFrog publish — is a plugin. Plugins are installed from the Jenkins UI or CLI. Each adds new job steps, UI sections, or integrations.

Install only what you need. Too many plugins slow Jenkins, cause conflicts, and create security exposure. Your roadmap needs: Git, Maven Integration, SonarQube Scanner, Artifactory, Docker Pipeline.

🎯

Freestyle Job vs Pipeline Job

Freestyle Job: configured entirely in the UI. Click-and-configure. Good for learning concepts. Not version-controlled. Pipeline Job: driven by a Jenkinsfile in your repo. Code-as-configuration. Version-controlled. Reproducible. This is what you use in production and what Day 9 covers.

Learn Freestyle today to understand Jenkins concepts. Use Pipeline/Jenkinsfile from Day 9 onward for everything real.

🌐

Webhook Trigger

A webhook is a HTTP POST that GitHub sends to Jenkins whenever code is pushed. Jenkins receives it, identifies which job to trigger, and starts the build. This is how "push code → build starts automatically" works. Requires Jenkins to be publicly reachable (your EC2 has a public IP).

The webhook is the heartbeat of your roadmap pipeline. Developer pushes code → GitHub fires webhook → Jenkins triggers → entire pipeline runs. Day 9 sets this up.

🔐

Jenkins Security

Jenkins out of the box is insecure — anonymous users can trigger builds. You must: create an admin account, enable authentication, use matrix-based security for fine-grained control, and use credentials manager (not plaintext) for passwords and tokens.

An unsecured Jenkins exposed to the internet will be found and abused within hours. Port 8080 on EC2 with no auth = remote code execution waiting to happen.

📋

Build Console Output

Every Jenkins build has a Console Output log — the full stdout/stderr of every step that ran. This is your primary debugging tool. Any build failure: first action is click Console Output. The error is always there.

New Jenkins users waste time looking at the job config when the build fails. The answer is always in Console Output. Read it completely before changing anything.

💥 Common Beginner Mistakes — Day 8

⚠ Jenkins Setup Mistakes
✗
Using t2.micro for Jenkins

Jenkins needs 2GB+ RAM for any real pipeline. t2.micro has 1GB. It will run Jenkins in idle but crash or hang the moment a Maven build starts. You will spend 2 hours debugging what is actually just an OOM kill.

✓ Fix: Always use t2.medium for Jenkins. Stop it when not studying to manage cost. This is non-negotiable.
✗
Skipping the security configuration

Jenkins with anonymous access enabled on a public EC2 means anyone on the internet can trigger your jobs, steal credentials, and execute code on your server. This happens within hours of exposure.

✓ Fix: In Manage Jenkins → Security: enable "Jenkins' own user database", disable "Allow users to sign up", set authorization to "Logged-in users can do anything". Never leave Jenkins open to anonymous users.
✗
Hardcoding passwords in build step shell scripts

echo "password=mypassword123" in a build step = that password is in every build log, visible to any Jenkins user with log access.

✓ Fix: Store all secrets in Jenkins Credentials Manager. Reference them in pipeline as credentials() binding. Never type a password in a build step.
✗
Not reading Console Output when a build fails

Jenkins users try random config changes when a build fails, hoping something works. 90% of the time the exact error is in the first 5 lines of Console Output.

✓ Fix: Build fails → click the build number → click Console Output → read the first red/error line. Fix that specific thing. No guessing.

⚡ Day 8 Quiz — Pass = 4/5
Q1. Why does Jenkins require at least 2GB RAM for a real CI pipeline?
A) Jenkins is written inefficiently
B) Jenkins runs the JVM, manages multiple build processes, and Maven itself is memory-intensive — they all compete for RAM simultaneously
C) AWS charges more for smaller instances
D) It only needs 512MB in production
Jenkins runs on the JVM (Java Virtual Machine) which itself uses 256–512MB. When a build runs, Maven launches another JVM process for the build. SonarQube analysis adds more. On a t2.micro with 1GB total, the OOM killer terminates processes randomly — causing mysterious build failures that are actually just memory starvation.
Q2. What is the difference between a Freestyle job and a Pipeline job in Jenkins?
A) Freestyle is faster; Pipeline is slower
B) Freestyle is configured in the UI (not version-controlled); Pipeline is driven by a Jenkinsfile in the repo (code as configuration, version-controlled)
C) Freestyle supports Docker; Pipeline does not
D) They are identical — just different names
Freestyle jobs: click-and-configure in the UI. Easy to start, but config lives only in Jenkins — if Jenkins is wiped, the config is lost. Pipeline jobs: driven by a Jenkinsfile committed to your GitHub repo. The pipeline definition is version-controlled alongside the code. If Jenkins is wiped and rebuilt, just point it at the repo and the pipeline recreates itself. Always use Pipeline/Jenkinsfile in production.
Q3. Where do you look first when a Jenkins build fails?
A) Jenkins system logs (/var/log/jenkins/jenkins.log)
B) The job configuration page
C) The build's Console Output — the full stdout/stderr of every step that ran
D) The GitHub repository for recent commits
Console Output (click build number → Console Output) shows every command executed and its output. The error is almost always clearly visible in red. Jenkins system logs are for Jenkins itself failing to start — not for build failures. This is the single most important Jenkins debugging habit.
Q4. What is the initial admin password in Jenkins and where is it stored?
A) It's "admin" by default
B) A randomly generated password stored at /var/lib/jenkins/secrets/initialAdminPassword on the server
C) It's emailed to you during installation
D) It's set during the apt install step
Jenkins generates a random password on first install and writes it to /var/lib/jenkins/secrets/initialAdminPassword. You read it with sudo cat and paste it into the browser to unlock Jenkins. Once you create your admin user through the setup wizard, this file is no longer needed — but the admin user you create IS your permanent login.
Q5. Why should you store GitHub credentials in Jenkins Credentials Manager instead of in build scripts?
A) Build scripts don't support environment variables
B) Credentials Manager encrypts and masks secrets — they never appear in build logs or are visible to users browsing job configs
C) Credentials Manager is required for GitHub webhooks to work
D) Jenkins can't access GitHub without Credentials Manager
Jenkins Credentials Manager stores secrets encrypted on disk. When used in a pipeline via credentials() binding, the secret value is masked in Console Output — shown as ****. If you hardcode a password in a shell script, it appears in plaintext in every build log, visible to any Jenkins user with read access to that job. Credentials Manager is not optional — it's mandatory for any production Jenkins instance.

🧠 Core Concepts

📋

Declarative Pipeline Syntax

Jenkinsfile uses a Groovy-based DSL (Domain Specific Language). Declarative pipelines use a structured format: pipeline → agent → stages → stage → steps. Every element has a specific position. Put a step outside steps and the whole file breaks.

Always use Declarative syntax (not Scripted). Declarative is easier to read, has better validation, and is the modern standard. Scripted pipeline is legacy.

🤖

agent any

agent any means "run this pipeline on any available Jenkins agent (executor)." In your setup, the agent is the Jenkins server itself — same machine. In larger setups, agents are separate machines that do the actual build work while the Jenkins master just coordinates.

For now: agent any is correct. Week 5 introduces dedicated build agents when your pipeline needs specific tools (Maven, Docker, SonarQube scanner) in isolated environments.

🎭

Stages and Stage View

Each stage is a logical step in the pipeline. Checkout → Build → Test → Archive. In the Jenkins Pipeline Stage View plugin, each stage shows as a column with green (success) or red (failed) indicators. You see at a glance which stage failed and how long each took.

Stages are not just visual. When a stage fails, all subsequent stages are skipped. This prevents broken code from reaching later stages (like deployment).

🔧

Maven in a Pipeline

Maven is a Java build tool. mvn clean package — clean removes old build files, package compiles + runs tests + creates a JAR file. The JAR (Java ARchive) is the deployable artifact. If any test fails, Maven exits with non-zero code → Jenkins marks stage as failed.

You don't need to understand Java to run Maven. Understand what mvn clean package does: compile + test + package. That's all your pipeline needs to know.

🌐

GitHub Webhook

A webhook is a URL GitHub calls when an event happens (a push). Your Jenkins URL + /github-webhook/ receives the POST. Jenkins sees it, identifies the matching job, and triggers a build. The flow: you push → GitHub fires HTTP POST to Jenkins → Jenkins starts pipeline.

This is what makes CI automatic. Without a webhook, you have to manually click "Build Now" every time. The webhook removes the human step — code pushed = build starts. Every CI system works this way.

📦

archiveArtifacts

After Maven builds the JAR, archiveArtifacts in the Jenkinsfile copies it to Jenkins's artifact storage. You can then download the JAR from the Jenkins UI, or pass it to downstream jobs (like JFrog upload in Day 12). This is how the build artifact travels through your pipeline.

In your roadmap: Jenkins archives the JAR → next day you push it to JFrog Artifactory instead of just storing it in Jenkins. archiveArtifacts is the starting point of artifact management.

💥 Common Beginner Mistakes — Day 9

⚠ Jenkinsfile + Pipeline Mistakes
✗
Writing the Jenkinsfile without testing the Maven build manually first

You write a complex Jenkinsfile, run the pipeline, and Maven fails. You don't know if it's a Jenkinsfile syntax issue or a Maven issue. Now you're debugging two things at once.

✓ Fix: ALWAYS manually verify mvn clean package works on the EC2 before writing a Jenkinsfile. Fix Maven issues first, then put it in the pipeline.
✗
Committing the target/ directory to Git

The target/ directory contains all compiled classes and JAR files — hundreds of MB of generated files that don't belong in version control. Committing them bloats your repo permanently.

✓ Fix: Add target/ to .gitignore before your first commit in any Maven project. This is the equivalent of node_modules/ in JavaScript projects.
✗
Forgetting the trailing slash in the webhook URL

http://IP:8080/github-webhook (no slash) returns 404. http://IP:8080/github-webhook/ (with slash) works correctly. This is a Jenkins-specific requirement and causes hours of confusion.

✓ Fix: Webhook Payload URL must end with /github-webhook/ — always with trailing slash.
✗
Using sh 'mvn ...' without a dir() block when pom.xml is in a subdirectory

Jenkins runs the sh step from the workspace root. If pom.xml is in devops-app/, running mvn from root fails with "The specified POM file was not found".

✓ Fix: Always use dir('devops-app') { sh 'mvn clean package' } to change into the correct directory before running Maven.

INT Q →"What is a Jenkinsfile and why is it better than configuring jobs in the Jenkins UI?"
SeniorAnswer: "A Jenkinsfile is a text file committed to the root of your source code repository that defines your entire Jenkins pipeline as code. It uses Jenkins's Declarative Pipeline syntax — a structured Groovy DSL with pipeline, stages, stage, and steps blocks. The key advantage over UI-configured Freestyle jobs is that the pipeline definition lives alongside the code it builds, so it's version-controlled, reviewed in pull requests, and reproducible. If your Jenkins server is wiped and rebuilt, you point a new Pipeline job at the same repo and the pipeline recreates itself from the Jenkinsfile — no manual reconfiguration. It also allows the pipeline to evolve with the code: when a developer adds a new tool or test framework, they update the Jenkinsfile in the same PR as the code change. INT Q →"Explain how a GitHub webhook triggers a Jenkins pipeline."
Answer: "When you push code to GitHub, GitHub can notify external services via a webhook — an HTTP POST request sent to a configured URL. In Jenkins, that URL is http://JENKINS_IP:8080/github-webhook/. I configure this URL in the GitHub repository settings under Webhooks. On the Jenkins side, I enable the 'GitHub hook trigger for GITScm polling' option in the Pipeline job configuration. When the developer pushes, GitHub immediately fires a POST to that Jenkins URL containing a JSON payload describing the push — which branch, which commit, who pushed. Jenkins receives it, identifies all jobs configured to watch that repository and branch, and starts those jobs automatically. This removes the human step from CI: push code, build starts, results are known within minutes — without anyone clicking anything."

🧠 Core Concepts — Maven

🔄

Maven Build Lifecycle

Maven has three built-in lifecycles. The default lifecycle is what you use: validate → compile → test → package → verify → install → deploy. Each phase runs all phases before it. Running mvn package automatically runs validate, compile, test first.

This is why mvn package fails when tests fail — test runs before package. You cannot skip a phase without explicitly telling Maven to (-DskipTests).

📋

pom.xml — Project Object Model

The pom.xml is Maven's config file. It defines: project coordinates (groupId, artifactId, version), dependencies (what libraries your code needs), plugins (what tools Maven uses during the build), and properties (variables used throughout the file).

Every Maven project has exactly one pom.xml at its root. Understanding it is understanding your entire build configuration in one file.

🏷

GAV Coordinates

Every Maven artifact is identified by three parts: GroupId (org.springframework.boot), ArtifactId (spring-boot-starter-web), Version (3.2.5). This is called the GAV coordinate. When you add a dependency to pom.xml, you give its GAV. Maven downloads it from Maven Central.

Interview: "What is a Maven dependency?" Answer: "A library identified by a GAV coordinate — groupId, artifactId, version. Maven downloads it from a registry and makes it available at compile time."

📦

Maven Local Repository (.m2)

Maven caches every downloaded dependency in ~/.m2/repository/. First build: slow (downloads from Maven Central). Subsequent builds: fast (uses local cache). On CI servers, the .m2 cache is often persisted between builds for speed.

On Jenkins: first build after fresh EC2 = 3-5 minutes downloading. Second build = 20-30 seconds. This is normal — not a performance problem with your pipeline.

🧪

Surefire Plugin — Test Execution

The Maven Surefire plugin runs unit tests (files ending in Test.java or beginning with Test). It generates XML reports in target/surefire-reports/. Jenkins reads these XML files to display test pass/fail graphs and individual test results.

The JUnit step in Jenkinsfile (junit '**/target/surefire-reports/*.xml') tells Jenkins to parse these XML files and display results in the UI — trend graphs, failed test details, test history.

🔌

Maven Plugins

Maven plugins add behaviour to the build lifecycle. compiler-plugin compiles Java. surefire-plugin runs tests. jar-plugin creates the JAR. You can add plugins like jacoco (code coverage), checkstyle (code style). Each plugin binds to a lifecycle phase and runs automatically when that phase is reached.

SonarQube (Day 11) is integrated via the sonar-maven-plugin. JFrog (Day 12) uses the artifactory-maven-plugin. Understanding plugins means understanding how every tool connects to your build.

⚠ Maven Mistakes
✗
Using mvn package in the Test stage instead of mvn test

mvn package runs tests AND packages. If you put it in Test stage and Package stage, you run tests twice — slow and redundant.

✓ Fix: Test stage: mvn test only. Package stage: mvn package -DskipTests (tests already ran).
✗
Not using mvn clean in CI — stale class files cause false positives

Without clean, if you rename a class, the old compiled .class file stays in target/. Tests may pass locally but fail in production because of stale code.

✓ Fix: In Jenkins always use mvn clean compile, mvn clean test, mvn clean package. The clean guarantees a fresh build from source.
✗
Hardcoding version numbers in Jenkinsfile instead of reading from pom.xml

If you put APP_VERSION='1.0.0' in Jenkinsfile and later update pom.xml to 1.1.0, they drift. The JAR name and the pipeline version disagree.

✓ Fix: Read version from pom.xml: sh "mvn help:evaluate -Dexpression=project.version -q -DforceStdout". Single source of truth.

INT Q →"What is the difference between mvn package and mvn install?"
SeniorAnswer: "Both compile, run tests, and create the JAR. ⚡ Day 10 Quiz — Pass = 4/5
Q1. You run mvn package. Which phases run in order?
A) package → test → compile
B) validate → compile → test → package
C) compile → package (test skipped)
D) Only the package phase runs
Maven lifecycle phases are sequential and cumulative. Running mvn package executes every phase before it: validate (check pom.xml), compile (compile source), test (run unit tests), then package (create JAR). This is why tests fail before packaging — test runs as part of the normal lifecycle.
Q2. What does the scope="test" in a pom.xml dependency mean?
A) The dependency only works on test servers
B) The dependency is available only for compiling and running tests — it is NOT included in the final JAR
C) The dependency version is for testing only
D) Maven skips this dependency during mvn test
Maven dependency scopes control when a dependency is available. scope=test: available at test compile time and test runtime, excluded from the final packaged JAR. This is correct for JUnit, Mockito — test tools that have no business being in your production artifact. Other scopes: compile (default, in JAR), provided (available but not bundled, e.g. servlet-api provided by Tomcat).
Q3. Where does Maven store downloaded dependencies on the server?
A) /opt/maven/dependencies/
B) ~/.m2/repository/ in the home directory of the user running Maven
C) Inside the project's target/ directory
D) /var/maven/cache/
~/.m2/repository/ is the local Maven repository. On Jenkins, the jenkins user's home is /var/lib/jenkins so the cache is at /var/lib/jenkins/.m2/repository/. This is why first builds are slow (downloads from Maven Central) and subsequent builds are fast (cache hit). Deleting .m2 forces a full re-download.
Q4. A test fails in Jenkins but passes locally. Most likely reason?
A) Jenkins has a bug
B) Local build has stale files in target/ OR the Jenkins environment differs (Java version, port in use, missing env var)
C) Maven version mismatch (usually)
D) The test is flaky and should be ignored
First check: does the local build use mvn clean test or just mvn test? Without clean, stale files may mask failures. Second: Jenkins runs as a different user (jenkins) with a different environment — no GUI, different locale, different ports. Spring Boot tests starting a server on port 8080 fail if Jenkins already uses 8080. Fix: use random ports in tests with @SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT).
Q5. What command shows all direct and transitive dependencies of your project?
A) mvn list-dependencies
B) mvn dependency:tree
C) mvn show-deps
D) cat pom.xml | grep dependency
mvn dependency:tree prints the full dependency hierarchy including transitive dependencies — libraries that your libraries depend on. This is essential for debugging classpath conflicts (two libraries bringing in different versions of the same dependency) and for understanding why your JAR is large.

🧠 Core Concepts — SonarQube

🔍

What SonarQube Measures

Bugs: code that will likely fail at runtime. Vulnerabilities: security issues (SQL injection risk, exposed secrets). Code Smells: maintainability issues (overly complex methods, duplicate code). Security Hotspots: code that needs manual security review.

Interview: "SonarQube performs static analysis — it reads your source code without running it and identifies defects, security risks, and maintainability problems before they reach production."

🚦

Quality Gate

A Quality Gate is a pass/fail threshold that your code must meet. Default gate: no new bugs, no new vulnerabilities, code coverage on new code ≥ 80%, duplicated lines ≤ 3%. If the gate fails, the Jenkins pipeline stops — broken quality code cannot proceed to JFrog or deployment.

The quality gate is the enforcement mechanism. Without it, SonarQube is just a report — code can still be deployed regardless of quality. The gate makes quality mandatory.

📊

Technical Debt

SonarQube calculates how long it would take a developer to fix all code smells. This is "technical debt." A project with 3 days of technical debt is manageable. 6 months of debt means the codebase is costly to change. Teams track this over time to prevent it accumulating.

When a manager asks "why does adding a feature take so long?" — technical debt is often the answer. SonarQube makes it visible and measurable.

📈

Code Coverage

What percentage of your source code is executed when your unit tests run. 80% coverage means 80% of your code lines are tested. SonarQube works with the JaCoCo plugin to display coverage. Low coverage = code that breaks silently because no test catches it.

Coverage without quality is meaningless — you can write bad tests that touch every line. But zero coverage is always a red flag. Target 70–80% meaningful coverage.

🔑

SonarQube Token

Jenkins authenticates to SonarQube using an analysis token — a long random string generated in SonarQube UI. This token is stored in Jenkins Credentials Manager and injected into the pipeline via withSonarQubeEnv(). Never hardcode tokens in Jenkinsfiles.

Same pattern as GitHub PAT: generate a token in the tool, store in Jenkins Credentials, reference by ID in pipeline. Credentials never in source code.

🔌

sonar-maven-plugin

SonarQube analysis is triggered from Maven: mvn sonar:sonar. This runs the sonar-maven-plugin which analyzes your compiled code, runs tests to collect coverage, and sends results to the SonarQube server. The server URL and token are passed as properties.

In your pipeline: the SonarQube stage runs mvn sonar:sonar inside a withSonarQubeEnv() block that automatically injects the server URL and token from Jenkins credentials.

🧠 Core Concepts — JFrog Artifactory

🏪

What Artifactory Is

Artifactory is a universal artifact repository manager. It stores build outputs (JARs, WARs, Docker images, npm packages, Python wheels) in one central location. It also acts as a proxy cache for public registries — Maven Central, Docker Hub — reducing external network calls and improving build speed.

Interview: "JFrog Artifactory is the artifact repository in the pipeline. After Jenkins builds and tests the code, the JAR is published to Artifactory with version metadata. Later stages pull the exact versioned artifact from Artifactory for deployment."

📚

Repository Types

Local: stores your own published artifacts (your JARs). Remote: proxy for external registries (Maven Central). Virtual: combines local + remote into one URL — your pom.xml points here, Maven gets everything from one place whether local or external.

In your roadmap: you create a local Maven repository for your JARs today. Day 13 adds a local Docker repository for images. A virtual repo combines both for downstream consumers.

🏷

Artifact Versioning

Every artifact in Artifactory has a path: com/devops/devops-app/1.0.0/devops-app-1.0.0.jar. This mirrors the GAV coordinate. With SNAPSHOT versions (1.0.0-SNAPSHOT), Artifactory appends a timestamp to each upload so old snapshots are preserved. Release versions (1.0.0) are immutable — uploaded once, never overwritten.

Never overwrite a release artifact. If you need to change it, bump the version number. Immutability means you can always reproduce exactly what was deployed on any date.

🔑

Build Info

When Jenkins publishes to Artifactory via the JFrog plugin, it attaches build information: Jenkins build number, git commit, who triggered the build, which tests passed. This creates full traceability — given any artifact in Artifactory, you can trace it back to the exact git commit and Jenkins build that produced it.

This traceability is mandatory for compliance and incident response. "Which version is running in production and what code does it contain?" — Artifactory build info answers this instantly.

🔐

Access Tokens

Artifactory uses user/password or access tokens for authentication. Create a dedicated CI user in Artifactory (not admin), grant it deploy permissions on your repository, generate a token for that user. Store the token in Jenkins Credentials Manager. Never use admin credentials for CI.

Same principle as IAM least privilege: the Jenkins user can deploy to your-repo, nothing else. If compromised, damage is limited.

⚡

rtMaven (JFrog Jenkins Plugin)

The JFrog Jenkins plugin provides rtMavenRun and rtMavenDeployer steps. rtMavenRun wraps mvn command and automatically collects build info. rtPublishBuildInfo sends that info to Artifactory. This is the production way to deploy from Jenkins — not mvn deploy with credentials in pom.xml.

Never put Artifactory credentials in pom.xml. The JFrog Jenkins plugin injects credentials at runtime from Jenkins Credentials Manager.

🧠 Docker Fundamentals — Complete from Zero

📦

Container vs Virtual Machine

A VM virtualises hardware — it runs a full operating system (kernel + libraries) inside another OS. A container shares the host kernel. It only isolates the filesystem, network, and process space. Containers start in milliseconds and use MBs of memory. VMs take minutes and use GBs.

Your Spring Boot JAR + its Java runtime + its dependencies = a container image. Run it anywhere with Docker installed — same result every time. This solves "it works on my machine."

🖼

Image vs Container

An image is a read-only blueprint — like a class definition. It contains your app code, runtime, libraries, config. A container is a running instance of an image — like an object. One image can run as many containers as you need. Deleting a container does not delete the image.

In your pipeline: Jenkins builds the image once. That same image runs in dev, test, staging, production — identical environment every time.

🍰

Image Layers + Caching

Each Dockerfile instruction creates a read-only layer. Layers are cached. If you change only your app code (COPY), Docker reuses all the earlier layers (FROM, RUN). Put frequently changing instructions last, rarely changing ones first — faster builds.

Wrong order: COPY code → RUN apt-get install → every code change reinstalls packages (slow). Right order: RUN installs → COPY code → code changes use cached install layer (fast).

📋

Key Dockerfile Instructions

FROM: base image to start from. WORKDIR: set working directory in container. COPY: copy files from host into image. RUN: execute command during build (creates layer). EXPOSE: document which port the app listens on. ENTRYPOINT: command that always runs when container starts.

ENTRYPOINT vs CMD: ENTRYPOINT defines the main process. CMD provides default arguments. For a Spring Boot JAR: ENTRYPOINT ["java", "-jar"] CMD ["app.jar"]

🔄

Container Lifecycle

docker run = pull image + create container + start it. docker stop = send SIGTERM (graceful shutdown). docker rm = delete stopped container. docker rmi = delete image. Stopped containers still exist and use disk. Always clean up.

In your pipeline: Jenkins creates a new image on every build. Tag it with the version. Push to JFrog. The pipeline does not run the container — Kubernetes does that in Week 3.

This is your private Docker registry. Instead of Docker Hub (public) or ECR (AWS-specific), JFrog keeps your images alongside your JARs — one tool managing all artifact types.

# eclipse-temurin is the official OpenJDK distribution. Alpine = minimal (small image)

WORKDIR /build

# Copy pom.xml first — so dependency downloads are cached separately from source code
# If you change only source code, this layer is cached and Maven doesn't re-download
COPY pom.xml .
COPY mvnw .
COPY .mvn .mvn
RUN ./mvnw dependency:go-offline -B
# dependency:go-offline downloads ALL dependencies into the image layer
# Subsequent builds reuse this layer unless pom.xml changes

# Now copy source code and build
COPY src ./src
RUN ./mvnw clean package -DskipTests -B

# ── Stage 2: Runtime image ──────────────────────────────────────────────────
# Use JRE (not JDK) for runtime — smaller, less attack surface
FROM eclipse-temurin:17-jre-alpine

WORKDIR /app

# Create non-root user — never run apps as root in containers
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Copy only the JAR from the builder stage — not the Maven build tools or source
COPY --from=builder /build/target/*.jar app.jar

# Document which port the Spring Boot app listens on

# Health check — Docker will monitor this endpoint
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \

INT Q →"What is a multi-stage Docker build and why do you use it for a Spring Boot app?"
SeniorAnswer: "A multi-stage build uses multiple FROM instructions in a single Dockerfile. The first stage uses a full JDK image to compile and package the Java application. The second stage starts fresh from a minimal JRE image — just enough to run Java, not build it — and copies only the compiled JAR from the builder stage. The result: your final Docker image contains only the JRE and the JAR, not Maven, not the JDK, not the source code. For a Spring Boot app this reduces the image from around 400MB using a JDK base to around 217MB using a JRE base. Beyond size, it improves security — the JDK has more attack surface than the JRE, and your source code is not shipped in the production image. In your pipeline, Jenkins builds this multi-stage image, pushes it to JFrog Docker Registry, and Kubernetes pulls it for deployment."
⚡ Day 13 Quiz — Pass = 4/5
Q1. What is the key difference between a Docker image and a container?
A) Images run on Windows; containers run on Linux
B) An image is a read-only blueprint; a container is a running instance of an image with a writable layer on top
C) Containers are smaller than images
D) They are identical — just different terminology
Image = read-only, layered filesystem snapshot. It never changes. Container = running process with a thin writable layer on top of the image's read-only layers. Multiple containers can run from the same image simultaneously. Deleting a container removes its writable layer but leaves the image intact.
Q2. Your pipeline fails with "docker: Got permission denied while trying to connect to the Docker daemon." What is the fix?
A) Run docker as root in the pipeline
B) Add the jenkins user to the docker group and restart the Jenkins service
C) Reinstall Docker
D) Change the pipeline to use sudo docker
The Docker daemon socket (/var/run/docker.sock) is owned by the docker group. The jenkins user needs to be in that group to use Docker without sudo. After: sudo usermod -aG docker jenkins, you MUST restart Jenkins (not reload) for the group change to take effect for the jenkins process: sudo systemctl restart jenkins.
Q3. Why should you put COPY pom.xml and RUN dependency:go-offline BEFORE COPY src in a Java Dockerfile?
A) Docker requires pom.xml before source code for compilation
B) Layer caching — if only source code changes, the dependency download layer is cached and doesn't re-execute
C) Security requirement for Java applications
D) Maven requires pom.xml to be in its own layer
Dockerfile layers are cached. If pom.xml hasn't changed, the layer containing dependency:go-offline (which downloads all dependencies) is reused from cache — saving 60+ seconds. If you COPY src first, every code change invalidates the pom.xml layer too and triggers a full dependency re-download. This ordering is the single most impactful Dockerfile optimisation for Java projects.
Q4. Docker push fails: "http: server gave HTTP response to HTTPS client." How do you fix it?
A) Enable HTTPS on Artifactory
B) Add Artifactory's IP:port to Docker's insecure-registries in /etc/docker/daemon.json
C) Use docker push --http flag
D) Change Artifactory port to 443
Docker defaults to HTTPS for any registry that isn't localhost. Your Artifactory is on HTTP port 8083. You must tell Docker to allow HTTP for that address. Add to /etc/docker/daemon.json: {"insecure-registries": ["ARTIFACTORY_IP:8083"]} then sudo systemctl restart docker. In production you would use HTTPS with a certificate — insecure-registries is for development/learning only.
Q5. What does ENTRYPOINT ["java", "-jar", "app.jar"] do in a Dockerfile, and why use exec form (JSON array)?
A) It documents which Java version is required
B) It defines the command that runs when the container starts. Exec form means signals (SIGTERM) go directly to java, enabling graceful shutdown
C) Shell form and exec form are identical in behaviour
D) ENTRYPOINT runs during build; CMD runs at start
ENTRYPOINT defines the main process. Shell form (ENTRYPOINT java -jar app.jar) wraps in /bin/sh -c — signals like SIGTERM (docker stop) don't reach the java process, preventing graceful shutdown. Exec form ["java", "-jar", "app.jar"] runs java directly as PID 1 — signals reach it properly, Spring Boot can shut down gracefully, connections drain cleanly.

☸️ Kubernetes — Introduction from Zero

❓

Why Kubernetes Exists

Docker solves "how do I package and run my app in a container." But: what happens when the container crashes? How do you run 10 copies for high traffic? How do you update them without downtime? Docker alone has no answer. Kubernetes orchestrates containers — it manages their lifecycle, scaling, networking, and updates across multiple servers.

K8s takes your Docker image and says: "I'll make sure this always runs, always scales, and always updates safely." This is why it comes after Docker in your roadmap.

🫛

Pod — Smallest Unit

A Pod is the smallest deployable unit in Kubernetes. It wraps one or more containers that share the same network and storage. Usually one container per Pod. Pods are ephemeral — when they crash, Kubernetes creates a new one. You never manage Pods directly — you manage Deployments which manage Pods.

Interview: "A Pod is a wrapper around one or more containers, sharing a network namespace. Kubernetes schedules Pods onto nodes and restarts them if they fail."

📦

Deployment — Manages Pods

A Deployment tells Kubernetes: "run 3 replicas of this Pod, using this Docker image, with these resource limits." If a Pod crashes, the Deployment controller notices the actual count (2) doesn't match desired (3) and creates a new Pod. Deployments also manage rolling updates.

You will always use a Deployment, never a bare Pod in production. Deployment = desired state declaration. K8s continuously reconciles actual state to match desired state.

🌐

Service — Stable Network Endpoint

Pods have dynamic IPs that change when Pods restart. A Service gives a stable DNS name and IP to a group of Pods. ClusterIP: only reachable inside the cluster. NodePort: exposed on each node's IP at a specific port. LoadBalancer: creates a cloud load balancer (AWS ALB).

The Service is how other Pods and external users reach your app. "spring-boot-service" → always routes to healthy spring-boot pods, regardless of which pods are alive.

🗂

Namespace

A Namespace is a virtual cluster within a physical cluster. Isolates resources by team or environment. default (what you use now), kube-system (K8s internals), prod, dev, staging. Resources in different namespaces are isolated — Pod A in namespace dev cannot reach Pod B in namespace prod by default.

In your pipeline: Week 3 will deploy to different namespaces per environment. For now, use the default namespace.

🔑

kubectl — K8s CLI

kubectl is the command-line tool for every Kubernetes operation. kubectl get pods lists pods. kubectl apply -f file.yaml creates/updates resources. kubectl describe pod NAME shows events and status. kubectl logs POD_NAME shows container logs. kubectl exec -it POD -- bash opens a shell inside a pod.

All K8s interviews ask you to demonstrate kubectl commands. Know these 6 cold: get, describe, logs, exec, apply, delete.

Q1. What is the difference between mvn package and mvn deploy?
A) They are identical
B) mvn package creates JAR locally in target/; mvn deploy also uploads it to a remote repository (like JFrog Artifactory)
C) mvn deploy deploys the app to a server
D) mvn package skips tests; mvn deploy runs them
mvn package: compile + test + create JAR in target/. Stays local. mvn deploy: everything package does + uploads the artifact to the remote repository defined in pom.xml (or via the JFrog plugin). In your pipeline you use the JFrog plugin's rtMaven instead of plain mvn deploy, which gives better credential management and build info tracking.
Q2. SonarQube's waitForQualityGate step hangs for 5 minutes. What is the cause?
A) SonarQube token has expired
B) The SonarQube webhook pointing to Jenkins /sonarqube-webhook/ is not configured, so Jenkins never receives the quality gate result
C) Maven version is incompatible
D) Jenkins firewall blocks SonarQube
waitForQualityGate registers a listener endpoint on Jenkins at /sonarqube-webhook/. After analysis, SonarQube must POST the gate result to that URL. If the webhook is missing in SonarQube Administration → Webhooks, the POST never happens. Jenkins waits until timeout. Fix: create webhook in SonarQube pointing to http://JENKINS_IP:8080/sonarqube-webhook/ (trailing slash mandatory).
Q3. Why does the Dockerfile copy pom.xml before COPY src in the builder stage?
A) Docker requires this specific order for Java projects
B) Layer caching — if only source code changes, the dependency download layer is reused from cache, saving 60+ seconds per build
C) Maven cannot find pom.xml if placed after src
D) It reduces the final image size
Docker builds layers sequentially and caches each one. A cached layer is reused if neither the instruction nor any file it depends on has changed. pom.xml changes rarely (dependency updates). Source code changes constantly. By copying pom.xml first and running dependency:go-offline, that download layer is cached for all code-only changes — dropping rebuild time from 90s to 18s.
Q4. What is the difference between a Kubernetes Deployment and a Pod?
A) Pods are for stateful apps; Deployments for stateless apps
B) A Pod is a single running container group; a Deployment manages desired state — how many Pod replicas should run and restarts them if they fail
C) Deployments are only used in production
D) They are the same thing with different YAML structure
A Pod is ephemeral — when it crashes, it's gone. A Deployment is a controller that says "I want 2 replicas of this Pod spec running at all times." The Deployment controller continuously reconciles: actual Pods → desired count. If a Pod dies, the Deployment creates a new one. You always use Deployments, never bare Pods in production, because bare Pods are not self-healing.
Q5. What is the difference between livenessProbe and readinessProbe in Kubernetes?
A) They are identical — just different names
B) livenessProbe: if it fails, K8s restarts the container. readinessProbe: if it fails, K8s stops sending traffic to the Pod (but doesn't restart it)
C) livenessProbe checks disk; readinessProbe checks network
D) readinessProbe only runs at startup; livenessProbe runs continuously
livenessProbe: "is this container still alive?" If it fails repeatedly, K8s kills and restarts the container. Use for detecting deadlocks or crashes. readinessProbe: "is this container ready to serve traffic?" If it fails, the Pod is removed from the Service's load balancing — traffic goes only to healthy Pods. Use for warm-up time (Spring Boot takes 30s to start) — don't send traffic until it's ready, but don't restart it either.
Q6. You delete a Pod manually. What happens next in Kubernetes?
A) The Deployment is also deleted
B) The Deployment controller detects the actual replica count dropped below desired, and immediately creates a replacement Pod
C) Nothing — Kubernetes only manages Pods it created itself
D) An alert is sent but no automatic action is taken
The Deployment controller runs a continuous reconciliation loop: every few seconds it checks "do I have the desired number of healthy Pods?" When you delete a Pod, actual count drops below desired. The controller immediately creates a new Pod from the same spec. This is Kubernetes self-healing — the system automatically repairs itself without human intervention.
Q7. What is the purpose of a Kubernetes Service?
A) To build Docker images
B) To give a stable network endpoint (DNS name + IP) to a group of Pods, abstracting away their dynamic IPs
C) To monitor Pod health
D) To store persistent data
Pod IPs change every time a Pod is created. A Service has a stable ClusterIP and DNS name that never changes. It uses the selector to find Pods with matching labels and load-balances traffic across them. Other Pods and external users always connect to the Service, never directly to Pod IPs. When Pods are replaced (crash recovery, scaling, rolling update), the Service automatically routes to the new healthy Pods.
